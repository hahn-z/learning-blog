## 概念讲解

### 为什么需要最佳实践？

最佳实践不是教条，而是**踩坑经验的总结**。在 Vue 3 项目中，遵循一致的规范可以：

- 减少团队沟通成本
- 降低 Bug 发生率
- 提升代码可维护性
- 优化应用性能

### 核心维度

| 维度 | 关键实践 | 收益 |
|------|----------|------|
| 项目结构 | 按功能模块组织 | 可扩展、易定位 |
| 命名规范 | 统一命名风格 | 可读性 |
| 组件设计 | 单一职责 + Props Down/Events Up | 可复用、可测试 |
| 性能优化 | 懒加载 + 缓存 + 虚拟滚动 | 用户体验 |
| TypeScript | 完整类型覆盖 | 安全性 |
| 错误处理 | 统一错误边界 | 稳定性 |

### Props 和 Emits 类型化

```vue
<!-- src/components/UserCard.vue -->
<script setup lang="ts">
// ✅ Typed props with defaults
interface Props {
  user: {
    id: number
    name: string
    email: string
    avatar?: string
    role: 'admin' | 'user'
  }
  showActions?: boolean
  compact?: boolean
}

const props = withDefaults(defineProps<Props>(), {
  showActions: true,
  compact: false
})

// ✅ Typed emits
const emit = defineEmits<{
  click: [userId: number]
  delete: [userId: number]
  edit: [user: Props['user']]
}>()
</script>
```

### Composable 最佳实践

```ts
// src/composables/useRequest.ts
import { ref, type Ref } from 'vue'

// ✅ Accept Ref or primitive for flexibility
// ✅ Return typed reactive state
// ✅ Accept callback options
interface UseRequestOptions<T> {
  immediate?: boolean
  initialData?: T
  onSuccess?: (data: T) => void
  onError?: (error: Error) => void
}

interface UseRequestReturn<T> {
  data: Ref<T | undefined>
  loading: Ref<boolean>
  error: Ref<Error | null>
  execute: () => Promise<void>
  refresh: () => Promise<void>
}

export function useRequest<T>(
  requestFn: () => Promise<T>,
  options: UseRequestOptions<T> = {}
): UseRequestReturn<T> {
  const data = ref(options.initialData) as Ref<T | undefined>
  const loading = ref(false)
  const error = ref<Error | null>(null)

  async function execute() {
    loading.value = true
    error.value = null
    try {
      data.value = await requestFn()
      options.onSuccess?.(data.value as T)
    } catch (err: any) {
      error.value = err
      options.onError?.(err)
    } finally {
      loading.value = false
    }
  }

  if (options.immediate) {
    execute()
  }

  return { data, loading, error, execute, refresh: execute }
}
```

### 错误边界

```vue
<!-- src/components/ErrorBoundary.vue -->
<template>
  <slot v-if="!error" />
  <div v-else class="error-boundary">
    <h3>Something went wrong</h3>
    <p>{{ error.message }}</p>
    <button @click="reset">Try Again</button>
  </div>
</template>

<script setup lang="ts">
import { ref, onErrorCaptured } from 'vue'

const error = ref<Error | null>(null)

onErrorCaptured((err) => {
  error.value = err
  // Prevent propagation
  return false
})

function reset() {
  error.value = null
}
</script>
```

```
项目结构清晰 → 新成员快速上手

```

## 注意事项

| 事项 | 说明 | 建议 |
|------|------|------|
| 命名一致性 | 文件名、组件名、变量名 | 定规范、统一执行 |
| 组件粒度 | 太大→难维护，太小→碎片化 | 单一职责但不过度拆分 |
| Composable | 不是所有逻辑都要提取 | 跨组件复用时才提取 |
| 性能优化 | 过早优化是万恶之源 | 先做出来，再优化瓶颈 |
| TypeScript | 不要 any 滥用 | 渐进式类型化 |

## 避坑指南

- ❌ 组件 prop 用 `Object` / `Array` 类型 → ✅ 用 `interface` 或泛型精确描述
- ❌ `watch` 监听整个大对象 → ✅ 用 `getter` 只监听需要的属性
- ❌ `v-for` 没有 `:key` → ✅ 始终绑定唯一 key
- ❌ `v-if` 和 `v-for` 同时用 → ✅ 先 filter 再 v-for，或用 `<template v-for>` + 内部 `v-if`
- ❌ 直接修改 props → ✅ emit 事件让父组件修改
- ❌ 全局状态满天飞 → ✅ 仅跨组件共享的状态放 Pinia

### v1 — 无规范
```ts

  data() { return { items: [], loading: false } },
  methods: { async fetch() { this.items = await axios.get('/api/items') } }
```

```ts
// Composition API but no composable extraction
```

```ts
// Typed composable + proper structure + performance optimization
const { data: items, loading, execute: fetch } = useRequest(() => itemApi.getList(), { immediate: true })
```
