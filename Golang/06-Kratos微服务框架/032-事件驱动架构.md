# 032-事件驱动架构

> 难度：⭐⭐⭐⭐⭐（高级架构篇）
> 适用：微服务架构师、后端高级工程师、分布式系统开发者
> 前置知识：Kratos基础、gRPC、消息队列基础、分布式系统概念

---

## 一、概念讲解

### 1.1 什么是事件驱动架构？

事件驱动架构（EDA, Event-Driven Architecture）是一种以**事件的产生、检测和消费**为核心的架构模式。服务之间通过事件进行异步通信，而非同步调用。

**核心要素：**

| 要素 | 说明 |
|------|------|
| 事件（Event） | 系统中发生的有意义的状态变化 |
| 事件生产者 | 产生并发布事件的组件 |
| 事件消费者 | 订阅并处理事件的组件 |
| 事件总线/消息代理 | 传递事件的基础设施（Kafka、NATS等） |

### 1.2 领域事件

领域事件是DDD中的核心概念，表示**领域中有意义的业务事件**：

```
订单创建 → OrderCreatedEvent
支付完成 → PaymentCompletedEvent  
商品库存扣减 → StockDeductedEvent
```

### 1.3 最终一致性

事件驱动架构中，**不追求强一致性**，而是保证最终一致性：

- 订单创建后，库存扣减可能延迟几百毫秒
- 系统保证在一定时间后，所有服务的数据最终一致
- 通过补偿事务处理异常情况

### 1.4 Saga模式

Saga是管理分布式事务的模式，分为两种：

- **编排式（Choreography）**：各服务通过事件自行协调
- **控制式（Orchestration）**：中心协调器控制整个流程

## 三、完整Go代码

### 3.1 领域事件定义

```go
// internal/event/events.go - Domain event definitions
package event

import "time"

// BaseEvent provides common fields for all domain events
type BaseEvent struct {
	EventID   string    `json:"event_id"`
	EventType string    `json:"event_type"`
	Timestamp time.Time `json:"timestamp"`
	AggregateID string  `json:"aggregate_id"`
}

// OrderCreatedEvent is raised when a new order is placed
type OrderCreatedEvent struct {
	BaseEvent
	UserID    string  `json:"user_id"`
	ProductID string  `json:"product_id"`
	Quantity  int32   `json:"quantity"`
	Amount    float64 `json:"amount"`
}

// PaymentCompletedEvent is raised when payment succeeds
type PaymentCompletedEvent struct {
	BaseEvent
	OrderID string  `json:"order_id"`
	Amount  float64 `json:"amount"`
	Method  string  `json:"method"`
}

// StockDeductedEvent is raised when inventory is deducted
type StockDeductedEvent struct {
	BaseEvent
	ProductID string `json:"product_id"`
	Quantity  int32  `json:"quantity"`
}

// StockDeductionFailedEvent is raised when stock is insufficient
type StockDeductionFailedEvent struct {
	BaseEvent
	OrderID   string `json:"order_id"`
	ProductID string `json:"product_id"`
	Reason    string `json:"reason"`
}
```

### 3.2 事件发布器

```go
// internal/event/publisher.go - Event publisher with Kafka backend
package event

import (
	"context"
	"encoding/json"
	"fmt"
	"log"

	"github.com/segmentio/kafka-go"
)

// Publisher handles event publishing to Kafka
type Publisher struct {
	writer *kafka.Writer
}

// NewPublisher creates a Kafka-based event publisher
func NewPublisher(brokers []string, topic string) *Publisher {
	return &Publisher{
		writer: &kafka.Writer{
			Addr:         kafka.TCP(brokers...),
			Topic:        topic,
			Balancer:     &kafka.LeastBytes{},
			RequiredAcks: kafka.RequireAll, // Ensure durability
			MaxAttempts:  3,
		},
	}
}

// Publish sends an event to Kafka
func (p *Publisher) Publish(ctx context.Context, event interface{}) error {
	data, err := json.Marshal(event)
	if err != nil {
		return fmt.Errorf("marshal event: %w", err)
	}

	// Extract event type and aggregate ID for message key
	base := extractBaseEvent(event)
	key := base.AggregateID

	msg := kafka.Message{
		Key:   []byte(key),
		Value: data,
		Headers: []kafka.Header{
			{Key: "event_type", Value: []byte(base.EventType)},
		},
	}

	if err := p.writer.WriteMessages(ctx, msg); err != nil {
		return fmt.Errorf("publish event: %w", err)
	}

	log.Printf("Published event: type=%s aggregate=%s", base.EventType, base.AggregateID)
	return nil
}

// Close shuts down the publisher
func (p *Publisher) Close() error {
	return p.writer.Close()
}

func extractBaseEvent(e interface{}) BaseEvent {
	type eventer interface{ GetBaseEvent() BaseEvent }
	if evt, ok := e.(eventer); ok {
		return evt.GetBaseEvent()
	}
	return BaseEvent{EventType: "unknown"}
}
```

### 3.3 Saga编排实现

```go
// internal/saga/order_saga.go - Order creation saga (choreography style)
package saga

import (
	"context"
	"encoding/json"
	"fmt"
	"log"

	"github.com/segmentio/kafka-go"

	"myapp/internal/event"
)

// OrderSaga coordinates the order creation flow
type OrderSaga struct {
	stockDeductPublisher *event.Publisher
	compensationPublisher *event.Publisher
}

// NewOrderSaga creates a new saga coordinator
func NewOrderSaga(stockPub, compPub *event.Publisher) *OrderSaga {
	return &OrderSaga{
		stockDeductPublisher: stockPub,
		compensationPublisher: compPub,
	}
}

// HandleOrderCreated processes OrderCreatedEvent
// Saga step 1: Deduct stock after order is created
func (s *OrderSaga) HandleOrderCreated(ctx context.Context, msg kafka.Message) error {
	var evt event.OrderCreatedEvent
	if err := json.Unmarshal(msg.Value, &evt); err != nil {
		return fmt.Errorf("unmarshal: %w", err)
	}

	log.Printf("Saga: processing order %s, deducting stock for product %s",
		evt.AggregateID, evt.ProductID)

	// Publish stock deduction command
	deductEvent := event.StockDeductedEvent{
		BaseEvent: event.BaseEvent{
			EventID:     fmt.Sprintf("evt-stock-%s", evt.AggregateID),
			EventType:   "stock.deduct.requested",
			Timestamp:   evt.Timestamp,
			AggregateID: evt.ProductID,
		},
		ProductID: evt.ProductID,
		Quantity:  evt.Quantity,
	}

	return s.stockDeductPublisher.Publish(ctx, deductEvent)
}

// HandleStockDeductionFailed processes stock failure - triggers compensation
func (s *OrderSaga) HandleStockDeductionFailed(ctx context.Context, msg kafka.Message) error {
	var evt event.StockDeductionFailedEvent
	if err := json.Unmarshal(msg.Value, &evt); err != nil {
		return fmt.Errorf("unmarshal: %w", err)
	}

	log.Printf("Saga: stock deduction failed for order %s, compensating", evt.OrderID)

	// Publish compensation: cancel order
	compensateEvent := event.BaseEvent{
		EventID:     fmt.Sprintf("evt-compensate-%s", evt.OrderID),
		EventType:   "order.cancel.requested",
		Timestamp:   evt.Timestamp,
		AggregateID: evt.OrderID,
	}

	return s.compensationPublisher.Publish(ctx, compensateEvent)
}
```

### 3.4 事件消费者框架

```go
// internal/event/subscriber.go - Event subscriber with idempotency
package event

import (
	"context"
	"encoding/json"
	"fmt"
	"log"
	"sync"

	"github.com/segmentio/kafka-go"
)

// Handler processes a specific event type
type Handler func(ctx context.Context, msg kafka.Message) error

// Subscriber manages event consumption from Kafka
type Subscriber struct {
	reader  *kafka.Reader
	handlers map[string]Handler
	seen    map[string]bool // Simple idempotency check
	mu      sync.Mutex
}

// NewSubscriber creates a Kafka consumer group subscriber
func NewSubscriber(brokers []string, topic, groupID string) *Subscriber {
	return &Subscriber{
		reader: kafka.NewReader(kafka.ReaderConfig{
			Brokers:  brokers,
			Topic:    topic,
			GroupID:  groupID,
			MinBytes: 1,
			MaxBytes: 10e6, // 10MB max per fetch
		}),
		handlers: make(map[string]Handler),
		seen:     make(map[string]bool),
	}
}

// Register adds a handler for a specific event type
func (s *Subscriber) Register(eventType string, handler Handler) {
	s.handlers[eventType] = handler
}

// Start begins consuming events (blocking call)
func (s *Subscriber) Start(ctx context.Context) error {
	for {
		msg, err := s.reader.ReadMessage(ctx)
		if err != nil {
			if ctx.Err() != nil {
				return nil // Context cancelled, shutdown
			}
			log.Printf("Read error: %v", err)
			continue
		}

		// Extract event type from header
		eventType := getHeader(msg, "event_type")

		// Idempotency check using message key + offset
		dedupeKey := fmt.Sprintf("%s-%d-%d", msg.Topic, msg.Partition, msg.Offset)
		s.mu.Lock()
		if s.seen[dedupeKey] {
			s.mu.Unlock()
			continue // Already processed
		}
		s.seen[dedupeKey] = true
		s.mu.Unlock()

		// Dispatch to handler
		handler, ok := s.handlers[eventType]
		if !ok {
			log.Printf("No handler for event type: %s", eventType)
			continue
		}

		if err := handler(ctx, msg); err != nil {
			log.Printf("Handler error for %s: %v", eventType, err)
			// In production: send to DLQ (Dead Letter Queue)
		}
	}
}

// Close shuts down the subscriber
func (s *Subscriber) Close() error {
	return s.reader.Close()
}

func getHeader(msg kafka.Message, key string) string {
	for _, h := range msg.Headers {
		if h.Key == key {
			return string(h.Value)
		}
	}
	return ""
}
```

## 五、注意事项

| 注意点 | 说明 | 影响 |
|--------|------|------|
| 幂等性 | 同一事件可能被消费多次，处理逻辑必须幂等 | 重复扣款/重复发货 |
| 消息顺序 | Kafka保证分区内有序，跨分区无序 | 需要合理的分区键设计 |
| 事件丢失 | 消费者crash可能丢消息，需确认机制 | 数据不一致 |
| 事件Schema演进 | 事件结构会变化，需要兼容性策略 | 消费端反序列化失败 |
| 死信队列 | 处理失败的消息要进入DLQ，不能丢弃 | 运维必须 |
| 监控延迟 | 要监控事件消费延迟（lag） | 告警关键指标 |
| Saga补偿 | 补偿事务本身也可能失败，需要重试+人工介入 | 极端情况数据不一致 |

## 六、避坑指南

| ❌ 错误做法 | ✅ 正确做法 |
|------------|-----------|
| 事件中只传ID，消费者再查DB | 事件包含足够的数据，减少回查（事件充实） |
| 不做幂等，依赖消息只投递一次 | 每个处理逻辑都要幂等（去重表/唯一键） |
| 用同步调用模拟事件 | 真正的异步消息队列（Kafka/NATS），不要假事件 |
| 所有事件放一个Topic | 按领域/业务分Topic，避免消费组互相影响 |
| 忽略事件版本管理 | Schema Registry管理事件版本，保证向后兼容 |
| Saga没有补偿逻辑 | 每个正向操作都要有对应的补偿操作 |
| 不监控消费延迟 | 监控Kafka consumer lag，设告警阈值 |

## 十一、代码演进

### v1：同步调用模拟
```go
// v1: Synchronous chain - tight coupling, no resilience
func CreateOrder(ctx context.Context, req *OrderReq) error {
    order := createOrder(req)
    deductStock(order.ProductID, order.Quantity) // Fails? Inconsistent state
    processPayment(order.ID, order.Amount)        // Fails? Stock already deducted
    return nil
}
```

### v2：事件驱动 + 编排式Saga
```go
// v2: Event-driven with choreography saga
func CreateOrder(ctx context.Context, req *OrderReq) (*Order, error) {
    order := createOrder(req)
    publishEvent("order.created", order) // Fire event, let saga handle the rest
    return order, nil
}
// Stock service subscribes to order.created → deducts or publishes failure
// Payment service subscribes to stock.deducted → processes payment
```

### v3：控制式Saga + 事件溯源
```go
// v3: Orchestrated saga with event store
func (s *SagaOrchestrator) Execute(ctx context.Context, saga *SagaDef) error {
    for _, step := range saga.Steps {
        result, err := step.Execute(ctx)
        s.eventStore.Append(ctx, step.Event(result)) // Persist every step
        if err != nil {
            return s.compensate(ctx, saga, step) // Rollback with compensation
        }
    }
    return nil
}
```

**演进路线**：同步调用 → 事件驱动编排式 → 控制式Saga + 事件溯源
