# 025-API版本管理

> 难度：⭐⭐⭐（中级）
>
> 本文讲解 Kratos 微服务中 API 版本管理的完整策略，从 Proto 版本化到 URL/Header 版本控制，确保 API 平滑演进。

---

## 一、概念讲解

### 1.1 为什么需要版本管理？

API 是服务间和前后端的契约。业务演进不可避免地需要修改 API，版本管理确保：
- 老客户端不被破坏（向后兼容）
- 新功能可以安全推出
- 废弃流程可控、可追踪

### 1.2 版本管理策略对比

| 策略 | 方式 | 优点 | 缺点 |
|------|------|------|------|
| Proto 包版本 | `v1`/`v2` 包路径 | gRPC 原生支持 | 需要多套代码 |
| URL 版本化 | `/api/v1/users` | 直观，HTTP 友好 | gRPC 不适用 |
| Header 版本化 | `Accept: application/vnd.api.v2+json` | URL 干净 | 不够直观 |
| 字段演进 | Protobuf 兼容性规则 | 无需版本号 | 限制多 |

### 1.3 Protobuf 兼容性规则

以下变更是**兼容**的（不需要新版本）：
- 添加新字段（老客户端忽略）
- 添加新 enum 值
- 添加新 RPC 方法
- 将 `optional` 改为 `required`（单向）

以下变更是**破坏性**的（需要新版本）：
- 删除或重命名字段
- 修改字段编号
- 修改字段类型
- 删除 RPC 方法

## 三、完整代码

### v1: Proto 包版本化

```protobuf
// api/user/v1/user.proto - Stable v1 API
syntax = "proto3";
package api.user.v1;
option go_package = "your-project/api/user/v1";

service UserService {
  rpc GetUser(GetUserRequest) returns (GetUserReply);
  rpc ListUsers(ListUsersRequest) returns (ListUsersReply);
}

message GetUserRequest {
  int64 id = 1;
}

message GetUserReply {
  int64 id = 1;
  string username = 2;
  string email = 3;
}
```

```protobuf
// api/user/v2/user.proto - v2 with breaking changes
syntax = "proto3";
package api.user.v2;
option go_package = "your-project/api/user/v2";

service UserService {
  rpc GetUser(GetUserRequest) returns (GetUserReply);
  rpc ListUsers(ListUsersRequest) returns (ListUsersReply);
  // NEW: batch endpoint in v2
  rpc BatchGetUsers(BatchGetUsersRequest) returns (BatchGetUsersReply);
}

message GetUserRequest {
  // CHANGED: use string UUID instead of int64
  string id = 1;
}

message GetUserReply {
  string id = 1;
  string username = 2;
  // REMOVED: email field (privacy)
  // ADDED: profile fields
  UserProfile profile = 3;
  int64 created_at = 4;
}

message UserProfile {
  string display_name = 1;
  string avatar_url = 2;
}

message BatchGetUsersRequest {
  repeated string ids = 1;
}

message BatchGetUsersReply {
  repeated GetUserReply users = 1;
}
```

### v2: Kratos HTTP 路由版本化

```go
// internal/server/http.go - HTTP server with versioned routes
package server

import (
	"net/http"

	v1 "your-project/api/user/v1"
	v2 "your-project/api/user/v2"

	"github.com/go-kratos/kratos/v2/transport/http"
)

// NewHTTPServer creates HTTP server with versioned routes
func NewHTTPServer(
	v1Svc *service.UserServiceV1,
	v2Svc *service.UserServiceV2,
) *http.Server {
	srv := http.NewServer()

	// Register v1 routes under /v1/
	v1.RegisterUserServiceHTTPServer(srv, v1Svc)

	// Register v2 routes under /v2/
	v2.RegisterUserServiceHTTPServer(srv, v2Svc)

	// Custom version routing via middleware
	srv.HandlePrefix("/api/", versionMiddleware(srv))

	return srv
}

// versionMiddleware routes based on API version header or URL
func versionMiddleware(next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		// Check version from header
		version := r.Header.Get("X-API-Version")
		if version == "" {
			// Extract from URL path: /api/v2/users -> v2
			parts := strings.Split(strings.TrimPrefix(r.URL.Path, "/api/"), "/")
			if len(parts) > 0 && strings.HasPrefix(parts[0], "v") {
				version = parts[0]
			}
		}

		// Default to v1
		if version == "" {
			version = "v1"
		}

		// Add version to context
		ctx := context.WithValue(r.Context(), apiVersionKey{}, version)
		next.ServeHTTP(w, r.WithContext(ctx))
	})
}
```

### v3: 废弃管理 + 多版本适配器

```go
// internal/service/adapter.go - Adapter pattern for multi-version support
package service

import (
	"context"

	v1 "your-project/api/user/v1"
	v2 "your-project/api/user/v2"
)

// V1ServiceAdapter wraps v2 service to serve v1 API
// This allows gradual migration without maintaining two implementations
type V1ServiceAdapter struct {
	v2svc *UserServiceV2
}

// GetUser adapts v1 request to v2 and converts response back
func (a *V1ServiceAdapter) GetUser(ctx context.Context, req *v1.GetUserRequest) (*v1.GetUserReply, error) {
	// Convert v1 request (int64 id) to v2 request (string id)
	v2Req := &v2.GetUserRequest{
		Id: fmt.Sprintf("%d", req.Id),
	}

	// Call v2 implementation
	v2Resp, err := a.v2svc.GetUser(ctx, v2Req)
	if err != nil {
		return nil, err
	}

	// Convert v2 response to v1 response
	return &v1.GetUserReply{
		Id:       req.Id, // Keep original int64 id
		Username: v2Resp.GetUsername(),
		Email:    "", // v2 removed email, return empty
	}, nil
}
```

```go
// internal/service/deprecation.go - Deprecated endpoint tracking
package service

import (
	"context"
	"log"
	"sync/atomic"
)

// DeprecationTracker logs usage of deprecated endpoints
type DeprecationTracker struct {
	v1CallCount atomic.Int64
	removalDate string
}

// NewDeprecationTracker creates a tracker with removal date
func NewDeprecationTracker(removalDate string) *DeprecationTracker {
	return &DeprecationTracker{removalDate: removalDate}
}

// Track records a call to deprecated endpoint
func (t *DeprecationTracker) Track(ctx context.Context, endpoint string) {
	count := t.v1CallCount.Add(1)
	if count%100 == 0 {
		log.Printf("[DEPRECATION] %s called %d times, scheduled removal: %s",
			endpoint, count, t.removalDate)
	}
}
```

```protobuf
// api/user/v1/user.proto - Mark deprecated with annotations
syntax = "proto3";
package api.user.v1;

import "google/protobuf/descriptor.proto";

// Mark deprecated RPCs
// Deprecated: Use api.user.v2.UserService instead

  option deprecated = true;

  };
```

## 五、注意事项

| 事项 | 说明 |
|------|------|
| 兼容优先 | 能用字段演进就不用新版本 |
| 文档同步 | 每个版本有独立的 API 文档 |
| 废弃通知 | 提前 6 个月通知，给客户端迁移时间 |
| 监控用量 | 追踪旧版本调用量，决定下线时机 |
| Proto 兼容 | 使用 `buf breaking` 检测破坏性变更 |

---

## 六、避坑指南

| ❌ 错误做法 | ✅ 正确做法 |
|------------|-----------|
| 直接修改 v1 Proto 字段编号 | 创建 v2 包，保持 v1 不变 |
| 不标记废弃 | 使用 `option deprecated = true` |
| v1 v2 共享同一实现代码 | 适配器模式隔离 |
| 新版本删老字段 | 新版本用新字段号，老版本继续支持 |
| 不设过渡期 | 双版本并存至少 6 个月 |
| 没有版本路线图 | 提前规划版本时间表 |

## 七、练习题

🟢 **初级：** 创建 v1 和 v2 两个版本的 Proto 文件，保持向后兼容。

🟡 **中级：** 实现适配器模式，v1 API 内部调用 v2 实现并转换响应。

🔴 **高级：** 实现完整的版本废弃流程：标记废弃 → 追踪用量 → 自动通知 → 下线。

## 九、举一反三

| 场景 | 方案 |
|------|------|
| gRPC 版本管理 | Proto 包版本化，客户端按包路径选择 |
| REST API 版本 | URL 前缀 `/v1/` `/v2/` |
| 移动端适配 | 保留旧版本至少 2 个发布周期 |
| 网关层路由 | API Gateway 根据版本路由到不同服务 |
| 数据库兼容 | 新版本代码同时兼容新旧 Schema |
