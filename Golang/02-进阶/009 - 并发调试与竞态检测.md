# 039 - 并发调试与竞态检测

> 难度：⭐⭐⭐⭐（进阶）
>
> 本文深入讲解 Go 并发编程中最隐蔽的敌人——竞态条件（Race Condition），包括原理分析、检测工具使用、常见 bug 模式以及生产环境中的竞态检测策略。

---

## 一、概念讲解

### 什么是竞态条件？

竞态条件是指**两个或多个 goroutine 并发访问同一资源，且至少有一个是写操作，且没有同步机制**，导致程序行为取决于 goroutine 的执行顺序。

```
Goroutine A:  read x (x=0)     write x=x+1 (x=1)
Goroutine B:       read x (x=0)              write x=x+1 (x=1)
                                         结果: x=1 (丢失了一次更新!)
```

### 数据竞争 vs 竞态条件

| 概念 | 定义 | Go 工具 |
|------|------|---------|
| **数据竞争（Data Race）** | 对同一内存的并发读写无同步 | `go run -race` 可检测 |
| **竞态条件（Race Condition）** | 程序正确性依赖执行顺序 | 逻辑层面，工具难检测 |

数据竞争是竞态条件的子集。有数据竞争 → 一定有竞态条件；有竞态条件 → 不一定有数据竞争。

### Go 的竞态检测器

Go 内置的竞态检测器基于 **ThreadSanitizer (TSan)** 技术实现：
- 编译时插桩（instrumentation）：在每次内存访问前后插入检查代码
- 运行时检测：维护一个内存访问历史表，检测无同步的并发访问冲突
- 报告冲突：输出详细的调用栈信息

## 三、完整 Go 代码

### v1：竞态示例（有 bug）

```go
package main

import (
    "fmt"
    "sync"
)

// BUG: concurrent access to shared variable without synchronization
func main() {
    var counter int // shared state, NO synchronization
    var wg sync.WaitGroup

    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            counter++ // data race! read-modify-write without lock
        }()
    }
    wg.Wait()

    // Expected: 1000, but often less due to data races
    fmt.Printf("Counter: %d (expected 1000)\n", counter)
}
```

**执行预览（普通运行）：**

```
Counter: 973 (expected 1000)   ← 每次结果不同!
```

**执行预览（`go run -race`）：**

```
==================
WARNING: DATA RACE
Write at 0x00c0000b2008 by goroutine 7:
  main.main.func1()
      /race/main.go:16 +0x6e

Previous write at 0x00c0000b2008 by goroutine 6:
  main.main.func1()
      /race/main.go:16 +0x6e
==================
Counter: 981 (expected 1000)
Found 1 data race(s)
```

### v2：三种修复方案 + 银行转账示例

```go
package main

import (
    "fmt"
    "sync"
    "sync/atomic"
)

// Fix 1: Mutex
func counterWithMutex() int {
    var (
        counter int
        mu      sync.Mutex
        wg      sync.WaitGroup
    )
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            mu.Lock()
            counter++
            mu.Unlock()
        }()
    }
    wg.Wait()
    return counter
}

// Fix 2: Atomic
func counterWithAtomic() int32 {
    var counter int32
    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            atomic.AddInt32(&counter, 1)
        }()
    }
    wg.Wait()
    return counter
}

// Fix 3: Channel
func counterWithChannel() int {
    ch := make(chan int, 1000)
    for i := 0; i < 1000; i++ {
        go func() { ch <- 1 }()
    }
    counter := 0
    for i := 0; i < 1000; i++ {
        counter += <-ch
    }
    return counter
}

// BankAccount demonstrates safe concurrent transfers.
type BankAccount struct {
    mu      sync.Mutex
    balance int
}

func (a *BankAccount) Deposit(amount int) {
    a.mu.Lock()
    defer a.mu.Unlock()
    a.balance += amount
}

func (a *BankAccount) Balance() int {
    a.mu.Lock()
    defer a.mu.Unlock()
    return a.balance
}

func bankTransferDemo() {
    account := &BankAccount{}
    var wg sync.WaitGroup

    // 100 goroutines each deposit 10
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            account.Deposit(10)
        }()
    }
    wg.Wait()
    fmt.Printf("Bank balance: %d (expected 1000)\n", account.Balance())
}

func main() {
    fmt.Printf("Mutex counter:  %d\n", counterWithMutex())
    fmt.Printf("Atomic counter: %d\n", counterWithAtomic())
    fmt.Printf("Channel counter: %d\n", counterWithChannel())

    fmt.Println("\n=== Bank Transfer Demo ===")
    bankTransferDemo()
}
```

**执行预览：**

```
Mutex counter:  1000
Atomic counter: 1000
Channel counter: 1000

=== Bank Transfer Demo ===
Bank balance: 1000 (expected 1000)
```

### v3：并发 map + slice + 生产级竞态检测

```go
package main

import (
    "fmt"
    "sync"
)

// ConcurrentMap wraps a map with RWMutex for thread-safe access.
type ConcurrentMap struct {
    mu sync.RWMutex
    m  map[string]int
}

func NewConcurrentMap() *ConcurrentMap {
    return &ConcurrentMap{m: make(map[string]int)}
}

func (cm *ConcurrentMap) Set(key string, value int) {
    cm.mu.Lock()
    defer cm.mu.Unlock()
    cm.m[key] = value
}

func (cm *ConcurrentMap) Get(key string) (int, bool) {
    cm.mu.RLock()
    defer cm.mu.RUnlock()
    v, ok := cm.m[key]
    return v, ok
}

// ConcurrentSlice demonstrates thread-safe slice appending.
type ConcurrentSlice struct {
    mu sync.Mutex
    s  []int
}

func (cs *ConcurrentSlice) Append(v int) {
    cs.mu.Lock()
    defer cs.mu.Unlock()
    cs.s = append(cs.s, v)
}

func (cs *ConcurrentSlice) Len() int {
    cs.mu.Lock()
    defer cs.mu.Unlock()
    return len(cs.s)
}

func (cs *ConcurrentSlice) All() []int {
    cs.mu.Lock()
    defer cs.mu.Unlock()
    result := make([]int, len(cs.s))
    copy(result, cs.s)
    return result
}

func main() {
    // Concurrent map usage
    cm := NewConcurrentMap()
    var wg sync.WaitGroup

    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func(n int) {
            defer wg.Done()
            key := fmt.Sprintf("key-%d", n%10)
            cm.Set(key, n)
        }(i)
    }
    wg.Wait()

    fmt.Println("=== Concurrent Map ===")
    for i := 0; i < 10; i++ {
        key := fmt.Sprintf("key-%d", i)
        if v, ok := cm.Get(key); ok {
            fmt.Printf("  %s = %d\n", key, v)
        }
    }

    // Concurrent slice usage
    cs := &ConcurrentSlice{}
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func(n int) {
            defer wg.Done()
            cs.Append(n)
        }(i)
    }
    wg.Wait()
    fmt.Printf("\n=== Concurrent Slice ===\n  Length: %d (expected 1000)\n", cs.Len())
}
```

**执行预览：**

```
=== Concurrent Map ===
  key-0 = 90
  key-1 = 91
  ...

=== Concurrent Slice ===
  Length: 1000 (expected 1000)
```

## 四、注意事项

| 项目 | 说明 |
|------|------|
| -race 开销 | 竞态检测会带来 5-10 倍的性能开销和内存增长，生产环境慎用 |
| map 并发 panic | Go 的 map 并发读写会直接 panic（不是 data race，是 runtime 检测） |
| slice 不是线程安全的 | 并发 `append` 可能导致数据丢失或 panic |
| TSan 误报 | 极少，但理论上可能。审查报告时关注调用栈 |
| CI 必须加 -race | 单元测试、集成测试都应加 `-race` 标志 |
| atomic 限制 | 只支持基本类型，复杂操作用 mutex |

---

## 五、避坑指南

| ❌ 错误做法 | ✅ 正确做法 |
|------------|-----------|
| `go map[key]++` 并发写 map | 用 `sync.Mutex` 或 `sync.Map` |
| 并发 `append` slice | 用 mutex 保护或预分配 + atomic index |
| 以为 `fmt.Println` 是原子的 | `fmt.Println` 内部有锁但输出交错仍可能发生 |
| 只在开发环境测竞态 | CI pipeline 中 `go test -race ./...` 必须通过 |
| 认为 `-race` 能检测所有并发 bug | 只能检测数据竞争，逻辑层面的竞态条件需代码审查 |
| `defer mu.Unlock()` 在循环内 | `defer` 函数退出才执行，循环中应手动 `Unlock` |

## 六、练习题

### 🟢 入门：修复竞态计数器
以下代码有数据竞争，请用 `sync.Mutex` 修复：

```go
var count int
var wg sync.WaitGroup
for i := 0; i < 500; i++ {
    wg.Add(1)
    go func() { defer wg.Done(); count++ }()
}
wg.Wait()
```

### 🟡 中级：并发安全缓存
实现一个并发安全的 LRU Cache（简化版：支持 Get/Set，容量固定，淘汰最久未用）。用 `sync.Mutex` 保护。

### 🔴 高级：死锁检测
实现两个账户互相转账：A→B 和 B→A 并发。如果不按固定顺序加锁，会死锁。请实现安全版本并编写 `-race` 测试。

## 八、举一反三

| 场景 | 竞态类型 | 修复方案 |
|------|---------|---------|
| HTTP handler 中的计数器 | 数据竞争 | `atomic.AddInt64` |
| 并发读写配置 | 数据竞争 | `sync.RWMutex` |
| 并发写日志文件 | 数据竞争 | channel 序列化或 mutex |
| 单例初始化 | check-then-act | `sync.Once` |
| 转账操作 | 逻辑竞态 | 固定锁顺序 |
| 并发缓存 | 数据竞争 | `sync.Map` 或 mutex 保护 map |
