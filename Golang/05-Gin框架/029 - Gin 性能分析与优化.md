# Gin 性能分析与优化

> 难度：⭐⭐⭐⭐（高级）
> 前置知识：Gin 基础、Go 并发、HTTP 服务

## 一、概念讲解

Go 内置 `runtime/pprof` 和 `net/http/pprof`，可以实时分析 CPU、内存、goroutine 等性能瓶颈。Gin 应用集成 pprof 后，可以在线诊断性能问题，无需停服。

**pprof 能分析什么：**

| 类型 | 采集方式 | 分析内容 |
|------|----------|----------|
| CPU | 定时采样 | 哪些函数占用 CPU 最多 |
| Heap | 内存分配 | 哪些代码分配了最多内存 |
| Goroutine | 实时快照 | goroutine 数量和阻塞点 |
| Block | 阻塞采样 | channel/mutex 阻塞时长 |
| Mutex | 锁竞争 | 哪些锁竞争最激烈 |

**分析工具链：**
- `go tool pprof`：命令行交互分析
- 火焰图（Flame Graph）：可视化 CPU 分布
- pprof Web UI：浏览器查看
- `go-torch`：Uber 开源的火焰图工具

## 三、完整代码

### v1：基础集成

```go
// main.go - Basic pprof integration
package main

import (
	"log"
	"net/http"
	_ "net/http/pprof" // Auto-register /debug/pprof/*

	"github.com/gin-gonic/gin"
)

func main() {
	// Start pprof on separate port (security!)
	go func() {
		log.Println("pprof on :6060")
		log.Println(http.ListenAndServe("localhost:6060", nil))
	}()

	r := gin.Default()

	r.GET("/api/heavy", func(c *gin.Context) {
		// Simulate heavy computation
		result := make([]string, 0, 10000)
		for i := 0; i < 10000; i++ {
			result = append(result, "item_"+string(rune(i)))
		}
		c.JSON(200, gin.H{"count": len(result)})
	})

	r.GET("/api/leak", func(c *gin.Context) {
		// Simulate goroutine leak
		ch := make(chan struct{})
		go func() {
			<-ch // This goroutine will never finish
		}()
		c.JSON(200, gin.H{"status": "leaked"})
	})

	r.Run(":8080")
}
```

### v2：gin-contrib/pprof + 自定义分析端点

```go
// main.go - Using gin-contrib/pprof
package main

import (
	"fmt"
	"runtime"
	"time"

	"github.com/gin-contrib/pprof"
	"github.com/gin-gonic/gin"
)

// Custom memory stats endpoint
func memoryStatsHandler(c *gin.Context) {
	var m runtime.MemStats
	runtime.ReadMemStats(&m)

	c.JSON(200, gin.H{
		"alloc_mb":       fmt.Sprintf("%.2f", float64(m.Alloc)/1024/1024),
		"total_alloc_mb": fmt.Sprintf("%.2f", float64(m.TotalAlloc)/1024/1024),
		"sys_mb":         fmt.Sprintf("%.2f", float64(m.Sys)/1024/1024),
		"gc_count":       m.NumGC,
		"goroutines":     runtime.NumGoroutine(),
		"heap_objects":   m.HeapObjects,
	})
}

// Health check with performance awareness
func healthHandler(c *gin.Context) {
	var m runtime.MemStats
	runtime.ReadMemStats(&m)

	status := "healthy"
	if m.Alloc > 500*1024*1024 { // > 500MB
		status = "warning: high memory"
	}
	if numGR := runtime.NumGoroutine(); numGR > 10000 {
		status = fmt.Sprintf("warning: too many goroutines (%d)", numGR)
	}

	c.JSON(200, gin.H{
		"status":     status,
		"goroutines": runtime.NumGoroutine(),
		"memory_mb":  fmt.Sprintf("%.2f", float64(m.Alloc)/1024/1024),
	})
}

func main() {
	r := gin.Default()

	// Register pprof routes under /debug/pprof
	pprof.Register(r)

	// Custom monitoring endpoints
	r.GET("/debug/memory", memoryStatsHandler)
	r.GET("/health", healthHandler)

	// Business routes
	r.GET("/api/users", func(c *gin.Context) {
		c.JSON(200, gin.H{"users": []string{"Alice", "Bob"}})
	})

	r.Run(":8080")
}
```

### v3：生产级优化模板

```go
// middleware/performance.go - Performance-aware middleware
package middleware

import (
	"runtime"
	"time"

	"github.com/gin-gonic/gin"
)

// PerformanceMonitor tracks goroutine leaks and slow requests
func PerformanceMonitor() gin.HandlerFunc {
	return func(c *gin.Context) {
		start := time.Now()
		c.Next()
		duration := time.Since(start)

		// Log slow requests
		if duration > 500*time.Millisecond {
			// In production: send to monitoring system
			_ = c.Request.URL.Path
			_ = duration
		}

		// Check goroutine count
		if numGR := runtime.NumGoroutine(); numGR > 5000 {
			// Alert: potential goroutine leak
			_ = numGR
		}
	}
}
```

```go
// optimization/buffer_pool.go - sync.Pool for zero-alloc responses
package optimization

import (
	"bytes"
	"sync"
)

// Buffer pool to reduce GC pressure
var bufferPool = sync.Pool{
	New: func() interface{} {
		return new(bytes.Buffer)
	},
}

// GetBuffer retrieves a buffer from pool
func GetBuffer() *bytes.Buffer {
	buf := bufferPool.Get().(*bytes.Buffer)
	buf.Reset()
	return buf
}

// PutBuffer returns a buffer to pool
func PutBuffer(buf *bytes.Buffer) {
	bufferPool.Put(buf)
}
```

```go
// optimization/optimized_handler.go - Demonstrates key optimizations
package optimization

import (
	"strconv"
	"strings"
	"sync"

	"github.com/gin-gonic/gin"
)

// Pre-allocate response slice
var responsePool = sync.Pool{
	New: func() interface{} {
		s := make([]string, 0, 100)
		return &s
	},
}

func OptimizedHandler(c *gin.Context) {
	// BAD: allocates new slice every time
	// result := make([]string, 0, 100)

	// GOOD: reuse from pool
	resultPtr := responsePool.Get().(*[]string)
	result := *resultPtr
	result = result[:0] // Reset length

	defer func() {
		*resultPtr = result
		responsePool.Put(resultPtr)
	}()

	// BAD: string concatenation in loop
	// s := ""
	// for i := 0; i < 100; i++ { s += strconv.Itoa(i) }

	// GOOD: strings.Builder
	var sb strings.Builder
	for i := 0; i < 100; i++ {
		sb.WriteString("item_")
		sb.WriteString(strconv.Itoa(i))
		result = append(result, sb.String())
		sb.Reset()
	}

	c.JSON(200, gin.H{"count": len(result)})
}
```

## 五、注意事项

| 事项 | 说明 |
|------|------|
| 端口隔离 | pprof 必须在独立端口，不暴露公网 |
| CPU 采样 | 采样间隔建议 30s，太短不精确 |
| 内存分析 | 在 GC 后采样更准确：先 `debug.FreeOSMemory()` |
| Block/Mutex | 需要手动开启：`runtime.SetBlockProfileRate(1)` |
| 生产安全 | pprof 端口用防火墙限制访问 |
| 性能开销 | CPU profile 开销约 1-5%，可接受 |

## 六、避坑指南

| ❌ 常见错误 | ✅ 正确做法 |
|-------------|------------|
| pprof 和业务同端口 | 独立端口 + 防火墙隔离 |
| 线上开 CPU 采集不关 | 采集完即关闭，或使用持续 profiling 工具 |
| 只看 `top` 不看火焰图 | 火焰图能发现 `top` 看不到的问题 |
| 忽略 goroutine 泄漏 | 定期检查 goroutine 数量趋势 |
| 优化了不重要的函数 | 先 profiling 找瓶颈，再优化 |
| 字符串拼接用 `+` | 用 `strings.Builder` 减少 alloc |
