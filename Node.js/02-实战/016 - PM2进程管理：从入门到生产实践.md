# PM2进程管理：从入门到生产实践

> **难度：** ⭐⭐ 中级 | **预计阅读：** 15分钟
> 
> **前置知识：** Node.js基础、Express/Koa、Linux命令行基础

---

## 1. 概念讲解

PM2（Process Manager 2）是Node.js生产环境的进程管理器。它能让你告别 `nohup node app.js &` 的原始时代。

**为什么需要PM2？**

- Node.js单线程，一个未捕获异常就崩溃
- 手动管理多个服务太痛苦
- 需要自动重启、日志收集、负载均衡
- 部署流程需要零停机

**PM2核心能力：**

| 能力 | 说明 |
|------|------|
| 进程守护 | 崩溃自动重启 |
| 负载均衡 | Cluster模式利用多核 |
| 日志管理 | 统一收集、轮转 |
| 部署系统 | 内置deploy命令 |
| 监控面板 | monit / plus |
| 启动脚本 | 开机自启 |

## 3. 完整代码

### 3.1 示例应用（被管理的服务）

```javascript
// server.js - A sample Express app to be managed by PM2
const express = require('express');
const app = express();
const PORT = process.env.PORT || 3000;

// Simulate occasional crashes for demo purposes
let requestCount = 0;

app.get('/', (req, res) => {
  requestCount++;
  res.json({
    message: 'Hello from PM2-managed app',
    pid: process.pid,
    requestCount,
    uptime: process.uptime(),
    memory: process.memoryUsage()
  });
});

// Crash simulation endpoint (for testing auto-restart)
app.get('/crash', (req, res) => {
  res.json({ message: 'Crashing in 1 second...' });
  setTimeout(() => {
    throw new Error('Simulated crash for PM2 demo');
  }, 1000);
});

// Health check endpoint
app.get('/health', (req, res) => {
  res.json({ status: 'ok', pid: process.pid });
});

app.listen(PORT, () => {
  console.log(`Server started on port ${PORT}, PID: ${process.pid}`);
});
```

### 3.2 PM2 配置文件

```javascript
// ecosystem.config.js - PM2 process configuration file
module.exports = {
  apps: [
    {
      name: 'my-api',
      script: './server.js',
      
      // Instances: 'max' = CPU cores, or specific number
      instances: 'max',
      exec_mode: 'cluster',
      
      // Environment variables
      env: {
        NODE_ENV: 'development',
        PORT: 3000
      },
      env_production: {
        NODE_ENV: 'production',
        PORT: 8080
      },
      
      // Auto restart settings
      max_restarts: 10,
      restart_delay: 4000,
      min_uptime: '10s',
      max_memory_restart: '500M',
      
      // Log configuration
      log_date_format: 'YYYY-MM-DD HH:mm:ss Z',
      error_file: './logs/error.log',
      out_file: './logs/out.log',
      merge_logs: true,
      
      // Graceful shutdown
      kill_timeout: 5000,
      listen_timeout: 10000,
      
      // Watch mode (dev only, DO NOT use in production)
      watch: false,
      ignore_watch: ['node_modules', 'logs']
    },
    {
      name: 'my-worker',
      script: './worker.js',
      instances: 1,
      exec_mode: 'fork',
      env: {
        NODE_ENV: 'development'
      },
      env_production: {
        NODE_ENV: 'production'
      },
      cron_restart: '0 3 * * *'  // Restart daily at 3 AM
    }
  ],

  deploy: {
    production: {
      user: 'deploy',
      host: 'your-server.com',
      ref: 'origin/main',
      repo: 'git@github.com:yourname/repo.git',
      path: '/var/www/myapp',
      'pre-deploy-local': '',
      'post-deploy': 'npm install && pm2 reload ecosystem.config.js --env production',
      'pre-setup': ''
    }
  }
};
```

### 3.3 Worker 进程

```javascript
// worker.js - Background worker process
const { Worker, isMainThread } = require('worker_threads');

class BackgroundWorker {
  constructor() {
    this.isRunning = false;
    this.interval = null;
  }

  start() {
    this.isRunning = true;
    console.log(`Worker started, PID: ${process.pid}`);
    
    // Run periodic task every 30 seconds
    this.interval = setInterval(() => {
      this.processTask();
    }, 30000);
  }

  async processTask() {
    const startTime = Date.now();
    try {
      // Simulate background work (e.g., email sending, data cleanup)
      console.log(`[${new Date().toISOString()}] Processing background task...`);
      
      // Your actual background work here
      await new Promise(resolve => setTimeout(resolve, 1000));
      
      const duration = Date.now() - startTime;
      console.log(`Task completed in ${duration}ms`);
    } catch (err) {
      console.error('Task failed:', err.message);
    }
  }

  stop() {
    this.isRunning = false;
    if (this.interval) clearInterval(this.interval);
    console.log('Worker stopped gracefully');
  }
}

// Graceful shutdown handlers
const worker = new BackgroundWorker();
worker.start();

process.on('SIGINT', () => {
  worker.stop();
  process.exit(0);
});

process.on('SIGTERM', () => {
  worker.stop();
  process.exit(0);
});
```

### 3.4 代码演进

**v1 — 最简启动（开发用）：**
```bash
# Install and basic start
npm install -g pm2
pm2 start server.js --name my-api
```

**v2 — 配置文件 + Cluster模式：**
```bash
# Use ecosystem config with cluster mode
pm2 start ecosystem.config.js
pm2 reload my-api    # Zero-downtime restart
```

**v3 — 生产环境完整方案：**
```bash
# Full production setup
pm2 startup                    # Generate systemd script
pm2 start ecosystem.config.js --env production
pm2 save                       # Save process list
pm2 install pm2-logrotate      # Log rotation module
pm2 set pm2-logrotate:max_size 100M
pm2 set pm2-logrotate:retain 30
```

## 5. 注意事项

| 场景 | 注意事项 | 严重程度 |
|------|----------|----------|
| Cluster模式 | Session/WebSocket需要用Redis共享，内存不能共享 | 🔴 高 |
| 日志 | 默认日志不轮转，磁盘会满 | 🟡 中 |
| watch模式 | 生产环境绝对不要开watch | 🔴 高 |
| 内存泄漏 | 配置max_memory_restart作为兜底 | 🟡 中 |
| graceful shutdown | kill_timeout要大于业务处理最长耗时 | 🔴 高 |
| 0.1秒重启 | min_uptime设太小会导致反复重启 | 🟡 中 |
| 端口冲突 | 多实例cluster模式共享端口，fork模式要不同端口 | 🔴 高 |

## 6. 避坑指南

| ❌ 常见错误 | ✅ 正确做法 |
|-------------|------------|
| `pm2 start app.js` 开发也用PM2 | 开发用 `nodemon`，生产用PM2 |
| 不配置日志轮转 | 安装 `pm2-logrotate` 模块 |
| `pm2 restart` 停机重启 | 用 `pm2 reload` 零停机重启 |
| 不处理SIGTERM信号 | 注册process.on('SIGTERM')做清理 |
| Cluster模式用内存存Session | 用Redis/Memcached共享状态 |
| 忘记 `pm2 save` | 每次改完进程列表都save |
| 直接root运行PM2 | 创建专用用户运行 |
| `instances: 'max'` 在低配机器 | 根据CPU和内存设合理数量 |

## 9. 举一反三

| 场景 | 方案 | 关键配置 |
|------|------|----------|
| API服务需要高可用 | Cluster + nginx upstream | `instances: 'max'`, `exec_mode: 'cluster'` |
| 定时任务服务 | Fork模式 + cron_restart | `cron_restart: '0 */6 * * *'` |
| 微服务多进程 | 一个ecosystem配多个app | apps数组配置多个服务 |
| 容器化部署 | PM2 alpine镜像 | `keymetrics/pm2:latest-alpine` |
| 多服务器部署 | PM2 deploy + SSH | deploy配置块 |
| CI/CD集成 | pm2 reload不中断 | post-deploy hook中reload |
