服务限流算法（令牌桶/漏桶/滑动窗口）

难度标注

⭐⭐⭐（中级）

前置知识： 基础算法、Redis 基础、高并发概念

预估用时： 45-50 分钟

概念讲解

服务限流（Rate Limiting） 是通过限制单位时间内的请求数量来保护系统不被突发流量冲垮的流量控制技术。常见的三种限流算法各有特点：令牌桶允许突发流量，漏桶强制匀速输出，滑动窗口精确统计窗口内请求数。

现实类比： 三种限流就像三种地铁进站方式。令牌桶 = 售票机每隔固定时间生成一张票，有票就能进，票攒够了可以一波人一起进（允许突发）。漏桶 = 不管来多少人，安检口以固定速率放行，永远匀速。滑动窗口 = 看过去5分钟进了多少人，满了就等，精确但需要记录历史。

技术场景： 用户服务的 POST /api/users 注册接口需要限流（每秒最多 50 次），防止恶意注册。GET /api/users/{id} 查询接口限流 200 QPS。不同接口需要不同算法——注册用滑动窗口（精确统计），查询用令牌桶（允许突发）。

完整代码

1. 令牌桶算法

<?php
// app/Service/RateLimit/TokenBucket.php

declare(strict_types=1);

namespace App\Service\RateLimit;

use Hyperf\Redis\Redis;

class TokenBucket
{
    public function __construct(
        private Redis $redis
    ) {}

    /**
     * ✅ Token bucket algorithm
     * @param string $key Resource identifier
     * @param int $maxTokens Maximum tokens (burst capacity)
     * @param int $refillRate Tokens added per second
     * @param int $requestTokens Tokens needed per request (default 1)
     */
    public function allow(string $key, int $maxTokens, int $refillRate, int $requestTokens = 1): bool
    {
        $now = microtime(true);
        $redisKey = "ratelimit:token_bucket:{$key}";

        // ✅ Lua script for atomic token bucket operation
        $script = <<<'LUA'
local key = KEYS[1]
local max_tokens = tonumber(ARGV[1])
local refill_rate = tonumber(ARGV[2])
local requested = tonumber(ARGV[3])
local now = tonumber(ARGV[4])

-- Get current bucket state
local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens = tonumber(bucket[1]) or max_tokens
local last_refill = tonumber(bucket[2]) or now

-- ✅ Calculate tokens to add based on elapsed time
local elapsed = math.max(0, now - last_refill)
local tokens_to_add = elapsed * refill_rate
tokens = math.min(max_tokens, tokens + tokens_to_add)

-- Check if enough tokens available
if tokens >= requested then
    tokens = tokens - requested
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    redis.call('EXPIRE', key, math.ceil(max_tokens / refill_rate) + 1)
    return 1
else
    redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
    redis.call('EXPIRE', key, math.ceil(max_tokens / refill_rate) + 1)
    return 0
end
LUA;

        $result = $this->redis->eval($script, [$redisKey, $maxTokens, $refillRate, $requestTokens, $now], 1);
        
        return (bool)$result;
    }

    /**
     * 🟢 Get remaining tokens for monitoring
     */
    public function getRemainingTokens(string $key): int
    {
        $tokens = $this->redis->hGet("ratelimit:token_bucket:{$key}", 'tokens');
        return $tokens !== false ? (int)$tokens : 0;
    }
}

2. 漏桶算法

<?php
// app/Service/RateLimit/LeakyBucket.php

declare(strict_types=1);

namespace App\Service\RateLimit;

use Hyperf\Redis\Redis;

class LeakyBucket
{
    public function __construct(
        private Redis $redis
    ) {}

    /**
     * ✅ Leaky bucket algorithm - fixed rate output
     * @param string $key Resource identifier
     * @param int $capacity Bucket capacity (queue size)
     * @param float $leakRate Requests leaked (processed) per second
     */
    public function allow(string $key, int $capacity, float $leakRate): bool
    {
        $now = microtime(true);
        $redisKey = "ratelimit:leaky_bucket:{$key}";

        // ✅ Atomic Lua script for leaky bucket
        $script = <<<'LUA'
local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local leak_rate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

local bucket = redis.call('HMGET', key, 'water', 'last_leak')
local water = tonumber(bucket[1]) or 0
local last_leak = tonumber(bucket[2]) or now

-- ✅ Calculate leaked water (processed requests) since last check
local elapsed = math.max(0, now - last_leak)
local leaked = elapsed * leak_rate
water = math.max(0, water - leaked)

-- Check if bucket has room
if water < capacity then
    water = water + 1
    redis.call('HMSET', key, 'water', water, 'last_leak', now)
    redis.call('EXPIRE', key, math.ceil(capacity / leak_rate) + 1)
    return 1
else
    redis.call('HMSET', key, 'water', water, 'last_leak', now)
    return 0
end
LUA;

        $result = $this->redis->eval($script, [$redisKey, $capacity, $leakRate, $now], 1);
        return (bool)$result;
    }
}

3. 滑动窗口算法

<?php
// app/Service/RateLimit/SlidingWindow.php

declare(strict_types=1);

namespace App\Service\RateLimit;

use Hyperf\Redis\Redis;

class SlidingWindow
{
    public function __construct(
        private Redis $redis
    ) {}

    /**
     * ✅ Sliding window counter algorithm
     * @param string $key Resource identifier
     * @param int $limit Max requests in window
     * @param int $windowSeconds Window size in seconds
     */
    public function allow(string $key, int $limit, int $windowSeconds): bool
    {
        $now = time();
        $redisKey = "ratelimit:sliding_window:{$key}";

        // ✅ Lua script for precise sliding window
        $script = <<<'LUA'
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

-- ✅ Remove expired entries outside the window
local window_start = now - window
redis.call('ZREMRANGEBYSCORE', key, '-inf', window_start)

-- Count current entries in window
local count = redis.call('ZCARD', key)

if count < limit then
    -- Add current request with timestamp as score
    redis.call('ZADD', key, now, now .. ':' .. math.random(1000000))
    redis.call('EXPIRE', key, window + 1)
    return 1
else
    return 0
end
LUA;

        $result = $this->redis->eval($script, [$redisKey, $limit, $windowSeconds, $now], 1);
        return (bool)$result;
    }

    /**
     * 🟢 Get current count in window for monitoring
     */
    public function getCurrentCount(string $key, int $windowSeconds): int
    {
        $redisKey = "ratelimit:sliding_window:{$key}";
        $now = time();
        $this->redis->zRemRangeByScore($redisKey, '-inf', (string)($now - $windowSeconds));
        return $this->redis->zCard($redisKey);
    }
}

4. 用户服务限流应用

<?php
// app/Middleware/RateLimitMiddleware.php

declare(strict_types=1);

namespace App\Middleware;

use App\Service\RateLimit\TokenBucket;
use App\Service\RateLimit\SlidingWindow;
use Hyperf\HttpMessage\Exception\TooManyRequestsHttpException;
use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;
use Psr\Http\Server\MiddlewareInterface;
use Psr\Http\Server\RequestHandlerInterface;

class RateLimitMiddleware implements MiddlewareInterface
{
    // ✅ Per-route rate limit configuration
    private const ROUTE_LIMITS = [
        '/api/users' => ['algorithm' => 'sliding_window', 'limit' => 200, 'window' => 1],
        '/api/users/register' => ['algorithm' => 'sliding_window', 'limit' => 50, 'window' => 1],
        '/api/users/login' => ['algorithm' => 'token_bucket', 'max' => 100, 'refill' => 20],
    ];

    public function __construct(
        private TokenBucket $tokenBucket,
        private SlidingWindow $slidingWindow
    ) {}

    public function process(
        ServerRequestInterface $request,
        RequestHandlerInterface $handler
    ): ResponseInterface {
        $path = $request->getUri()->getPath();
        $config = $this->getMatchedConfig($path);

        if ($config === null) {
            return $handler->handle($request);
        }

        // ✅ Build rate limit key: path + client IP
        $clientIp = $request->getServerParams()['remote_addr'] ?? 'unknown';
        $key = "{$path}:{$clientIp}";

        $allowed = match ($config['algorithm']) {
            'token_bucket' => $this->tokenBucket->allow($key, $config['max'], $config['refill']),
            'sliding_window' => $this->slidingWindow->allow($key, $config['limit'], $config['window']),
            default => true,
        };

        if (!$allowed) {
            // 🔴 Rate limit exceeded
            throw new TooManyRequestsHttpException(
                null,
                'Rate limit exceeded. Please try again later.'
            );
        }

        return $handler->handle($request);
    }

    private function getMatchedConfig(string $path): ?array
    {
        // Exact match first
        if (isset(self::ROUTE_LIMITS[$path])) {
            return self::ROUTE_LIMITS[$path];
        }
        
        // Prefix match
        foreach (self::ROUTE_LIMITS as $route => $config) {
            if (str_starts_with($path, $route)) {
                return $config;
            }
        }
        return null;
    }
}

注意事项

约束 说明 违反后果
Lua 脚本保证原子性 多客户端并发时不能有竞态 计数不准，超限放行
Redis Key 要设过期 避免无限积累 内存泄漏
限流 Key 要合理 IP/用户ID/接口组合 限不住恶意用户或误杀正常用户
令牌桶参数要匹配业务 max 太大等于不限，太小正常用户受影响 限流失效或过度限制
滑动窗口用 Sorted Set 精确记录每个请求时间 用普通计数器不精确

避坑指南

典型错误 现象 正确做法
用 PHP 变量做计数器 多进程/协程间不共享 用 Redis + Lua 保证原子性
限流不区分用户 一个用户刷满影响所有人 Key 包含用户ID或IP
漏桶容量设太大 突发流量全部堆积 容量 = 平均速率 × 可接受延迟
窗口重置时刻突发 固定窗口算法在边界会 2x 流量 用滑动窗口代替固定窗口
返回 499/500 而非 429 客户端无法区分限流和故障 限流统一返回 429 Too Many Requests

      2. 🟡 进阶题

      1. 分层限流：实现 IP 级（1000 QPS）+ 用户级（100 QPS）+ 接口级（50 QPS）三层限流，全部通过才放行。

        2. 🔴 开放题

        1. 设计自适应限流：根据系统负载（CPU、响应时间）动态调整限流阈值，而不是固定值。

          2. 知识点总结

          限流算法
├── 令牌桶（Token Bucket）
│   ├── 匀速生成令牌
│   ├── 允许突发（桶内有积累令牌）
│   ├── 适合读多写少场景
│   └── Redis + Lua 原子实现
├── 漏桶（Leaky Bucket）
│   ├── 固定速率流出
│   ├── 不允许突发
│   ├── 适合需要匀速的场景
│   └── 队列 + 恒定速率
├── 滑动窗口（Sliding Window）
│   ├── 精确统计时间窗口
│   ├── Redis Sorted Set 实现
│   ├── 适合严格限流场景
│   └── 内存开销较大（记录每个请求）
├── 固定窗口（Fixed Window）
│   ├── 最简单但不精确
│   ├── 边界突发问题
│   └── 适合精度要求不高的场景
└── 选型指南
    ├── 允许突发 → 令牌桶
    ├── 匀速输出 → 漏桶
    └── 精确控制 → 滑动窗口

          举一反三

          场景 变种说明 关键差异
          单机限流 → 分布式限流 多实例共享限流状态 需要 Redis 集中计数
          IP 限流 → 用户限流 不同维度的 Key 用户需要先认证获取 ID
          固定阈值 → 动态阈值 根据负载调整 需要监控系统配合
