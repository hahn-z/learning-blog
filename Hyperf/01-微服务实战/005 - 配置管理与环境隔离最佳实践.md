# 配置管理与环境隔离最佳实践

> **难度：** ⭐⭐
> **前置知识：** Hyperf 入门（001）、PHP 数组操作
> **预估用时：** 40 分钟

---

## 一、概念讲解

**一句话定义：** 配置管理是将应用参数（数据库连接、缓存地址、第三方密钥等）从代码中分离，按环境（dev/staging/prod）动态加载的机制。

**现实类比：** 你有三套衣服——居家服（开发环境）、商务装（测试环境）、正装（生产环境）。衣服的样式不同，但穿衣服的流程一样。配置管理就是让你一键换装，而不是每次出门重新买衣服。

**技术场景：** 用户服务在本地开发用 MySQL root 无密码，测试环境用 Docker 内网数据库，生产环境用 RDS 云数据库 + 读写分离。如果没有环境隔离，每次部署都要改代码，极易出错。Hyperf 使用 `.env` 文件 + `config/` 目录实现优雅的多环境配置。

### 3.5 在代码中使用配置

```php
<?php
// ✅ app/Service/ConfigDemoService.php

declare(strict_types=1);

namespace App\Service;

use Hyperf\Context\ApplicationContext;
use Psr\Container\ContainerInterface;

class ConfigDemoService
{
    private ContainerInterface $container;

    public function __construct()
    {
        $this->container = ApplicationContext::getContainer();
    }

    // ✅ Read config values
    public function getConfig(): array
    {
        return [
            // 🟢 Using config() helper
            'db_host' => config('databases.default.host'),
            'jwt_expire' => config('user_service.jwt.expire'),
            'password_min' => config('user_service.password.min_length'),

            // 🟢 Using env() directly (discouraged in code, use config files)
            'app_env' => env('APP_ENV', 'production'),
        ];
    }

    // ✅ Dynamic config based on environment
    public function isDevEnvironment(): bool
    {
        return env('APP_ENV', 'production') === 'dev';
    }
}
```

## 五、注意事项

| 约束 | 说明 | 违反后果 |
|------|------|----------|
| `.env` 不入库 | 包含敏感信息 | 密码泄露到 Git |
| `env()` 只在配置文件用 | 代码中用 `config()` | 配置缓存后 `env()` 返回 null |
| 配置文件缓存 | 生产环境开启配置缓存 | 修改 `.env` 后需清除缓存 |
| 类型转换 | `env()` 返回 string | `(int)` 强制转换端口号等数字 |
| 默认值必须有 | `env('KEY', default)` | 部署时缺少变量导致 null 错误 |

## 六、避坑指南

| 典型错误 | 现象 | 正确做法 |
|----------|------|----------|
| ❌ 在业务代码中直接 `env()` | 配置缓存后获取不到值 | 在 `config/` 中用 `env()`，业务用 `config()` |
| ❌ `.env` 提交到 Git | 密钥泄露 | `.gitignore` 加 `.env`，提供 `.env.example` |
| ❌ 硬编码数据库密码 | 换环境要改代码 | 全部通过 `.env` 注入 |
| ❌ 忘记 `declare(strict_types)` | 类型不严格 | 每个文件顶部都加 |

## 七、练习题

🟢 **基础题 1：** 创建 `config/autoload/user_service.php`，添加用户头像默认配置（宽高、格式、大小限制）。

🟢 **基础题 2：** 编写 `.env.example` 模板文件，列出用户服务所有需要的环境变量并加注释。

🟡 **进阶题：** 实现一个 `ConfigRefreshCommand`，运行时从配置中心（如 Nacos/Apollo）拉取最新配置并热更新。

🔴 **开放题：** 设计一个多租户配置方案：不同租户有不同的功能开关和限制参数，配置存储在数据库中，通过缓存加速访问。

```
│   └── CI/CD 环境变量注入
└── 最佳实践
    ├── .env 不入库
    ├── env() 仅在 config/ 中使用
    ├── 提供默认值
    └── 生产环境配置缓存
```

### v3 🟢 多环境 + 配置验证

```php
<?php
// 🟢 Production-ready config with validation
declare(strict_types=1);

namespace App\Bootstrap;

use Hyperf\Contract\ConfigInterface;
use Hyperf\Context\ApplicationContext;
use RuntimeException;

class ConfigValidator
{
    // ✅ Validate critical config on startup
    public static function validate(): void
    {
        $config = ApplicationContext::getContainer()->get(ConfigInterface::class);
        
        $required = [
            'databases.default.host',
            'databases.default.database',
            'user_service.jwt.secret',
        ];

        foreach ($required as $key) {
            if (empty($config->get($key))) {
                throw new RuntimeException("Missing required config: {$key}");
            }
        }

        // 🔴 Security: Reject default JWT secret in production
        if (env('APP_ENV') === 'production') {
            $secret = $config->get('user_service.jwt.secret');
            if ($secret === 'change-me-in-production') {
                throw new RuntimeException('JWT secret must be changed in production!');
            }
        }
    }
}
```
